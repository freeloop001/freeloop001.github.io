## 什么是 RAG？

**RAG（Retrieval-Augmented Generation）** 即检索增强生成，是一种结合检索系统与大语言模型（LLM）的技术架构。

### 为什么需要 RAG？

纯 LLM 存在以下局限：
- **知识时效性差**：训练数据截止某日期，无法获取最新信息
- **幻觉问题**：模型可能生成看似合理但错误的内容
- **上下文长度限制**：无法一次性输入超长文档

RAG 通过"检索-增强"的流程解决这些问题：

```
用户问题 → 检索系统 → 相关文档 → LLM → 生成回答
```

### RAG 工作流程

1. **文档处理**：将长文本分块（chunk）并转换为向量
2. **向量存储**：存入向量数据库（如 Chroma、Milvus、FAISS）
3. **相似检索**：用户提问时，检索最相关的文档片段
4. **生成回答**：将检索结果作为上下文提供给 LLM

## 核心组件

### 1. 文本分块（Chunking）

将长文档分割成较小的片段，是 RAG 效果的关键。

```python
# 简单分块示例
text = "很长的文档内容..."
chunks = []
chunk_size = 500

for i in range(0, len(text), chunk_size):
    chunks.append(text[i:i + chunk_size])
```

**分块策略**：
- 固定大小分块
- 递归分块
- 按段落分块
- 语义分块（使用 LLM 判断）

### 2. 向量 Embedding

将文本转换为向量表示，常用模型：

| 模型 | 维度 | 特点 |
|------|------|------|
| text-embedding-ada-002 | 1536 | OpenAI 官方 |
| bge-base-zh-v1.5 | 768 | 中文效果优秀 |
| m3e-base | 768 | 中文、多语言 |

```python
from sentence_transformers import SentenceTransformer

model = SentenceTransformer('bge-base-zh-v1.5')
embeddings = model.encode(["用户问题"])
```

### 3. 向量数据库

存储和检索向量，主要选择：

| 数据库 | 类型 | 特点 |
|--------|------|------|
| **Chroma** | 轻量级 | 简单易用，适合原型 |
| **Qdrant** | 云原生 | 功能丰富，支持过滤 |
| **Milvus** | 企业级 | 大规模、高并发 |
| **FAISS** | 本地 | 速度快，适合小规模 |

```python
# Chroma 示例
import chromadb

client = chromadb.Client()
collection = client.create_collection("docs")

# 添加文档
collection.add(
    documents=["文档内容1", "文档内容2"],
    ids=["doc1", "doc2"]
)

# 检索
results = collection.query(
    query_texts=["用户问题"],
    n_results=3
)
```

### 4. 检索策略

- **相似度检索**：最常见，基于向量余弦相似度
- **混合检索**：结合关键词和向量检索
- **重排序（Re-ranking）**：用更精确的模型重新排序结果

## 框架对比

### 三大框架对比

| 特性 | LangChain | LlamaIndex | Haystack |
|------|-----------|------------|----------|
| **定位** | LLM 应用开发框架 | 数据索引与检索专家 | 端到端 RAG 流水线 |
| **学习曲线** | 中等 | 较低 | 较高 |
| **文档质量** | 中等 | 优秀 | 优秀 |
| **社区活跃度** | 非常高 | 高 | 中等 |
| **灵活性** | 高 | 中 | 高 |
| **适合人群** | 全栈开发者 | 专注数据检索 | 需要完整流水线 |

### LangChain

**优点**：
- 生态最丰富，集成几乎所有 LLM 和工具
- 社区庞大，示例丰富
- Agent 能力强大

**缺点**：
- API 变动频繁，版本兼容问题
- 文档有时不够清晰

### LlamaIndex

**优点**：
- 专注于数据索引，检索效果出色
- 文档详尽，学习资源丰富
- 与 LangChain 可无缝集成

**缺点**：
- Agent 能力较弱
- 灵活性略低于 LangChain

### Haystack

**优点**：
- 端到端解决方案，适合生产环境
- 支持多种检索器、阅读器
- 适合大规模企业级应用

**缺点**：
- 学习曲线较陡
- 社区相对较小

---

## 如何选择？

- **快速原型 / 个人项目**：LangChain + LlamaIndex 组合
- **专注检索效果**：LlamaIndex
- **企业级生产环境**：Haystack 或自建 pipeline
- **结合使用**：LlamaIndex 负责索引，LangChain 负责编排

---

## 总结

RAG 是当前解决 LLM 知识局限的主流方案。三大框架各有优劣，关键是根据项目需求和团队熟悉度选择。对于大多数场景，LangChain 和 LlamaIndex 的组合已经足够强大。
